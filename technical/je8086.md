# JE-8086 (Roland JP-8000/8080) Technical Information

All technical information we discovered on the architecture of the Roland JP-8000 and JP-8080 synthesizers is documented here. Unlike the other synths in the Gearmulator project, the JP-8000 uses a completely different processor architecture вҖ” a Hitachi H8S microcontroller paired with custom Roland ESP ASICs (4 on the JP-8000, 5 on the JP-8080) вҖ” rather than the Motorola DSP56300 family found in the Waldorf, Access, and Clavia instruments.

## Introduction

Released in 1996, the Roland JP-8000 is one of the most iconic virtual analog synthesizers ever produced. It was among the first synthesizers to use digital signal processing to emulate the sound of classic analog instruments, and it introduced several ground-breaking features that fundamentally shaped electronic music production.

Most famously, the JP-8000 introduced the **Supersaw** waveform вҖ” a stack of seven detuned sawtooth waves that became the defining sound of late-1990s and 2000s trance music. The Supersaw's rich, shimmering character was unlike anything available from other synthesizers at the time, and it quickly became the go-to lead and pad sound for producers worldwide.

The JP-8000's impact on the trance music scene cannot be overstated. Darude's 1999 hit "Sandstorm" вҖ” one of the most recognizable electronic tracks ever produced вҖ” was created almost entirely on the JP-8080 (the rackmount version). The track is named after the JP-8080's first factory preset, which is literally called "Sandstorm". Other artists like Ferry Corsten, Paul van Dyk, TiГ«sto, Armin van Buuren, and ATB relied heavily on the JP-8000 series for their signature sounds throughout the late 1990s and early 2000s.

The JP-8000 (49-key keyboard) provides 8 voices of polyphony, while the JP-8080 (2U rackmount) expands this to **10 voices** with an additional fifth ESP ASIC. Both feature two oscillators per voice, a resonant multi-mode filter, two LFOs, comprehensive envelope generators, built-in effects (chorus, delay), and Roland's unique Motion Control system for recording knob movements.

## Hitachi H8S Microcontroller

The JP-8000 uses a **Hitachi H8S/2655** (H8S/SX16 family) 16-bit microcontroller running at **16 MHz**. This is a CISC processor with a 16-bit data bus, unlike the MC68331 used by the Waldorf and Clavia synths.

The H8S handles all system management tasks:
- Boot and initialization
- Front panel scanning (buttons, faders, LEDs)
- LCD display management
- MIDI communication
- Patch/performance storage and recall
- Loading and managing DSP programs on the ESP ASICs

### Memory Map

| Address Range | Size | Contents |
|--------------|------|----------|
| `$000000 - $07FFFF` | 512 KB | ROM (keyboard) |
| `$000000 - $0DFFFF` | 896 KB | ROM (rack, larger) |
| `$200000 - $23FFFF` | 256 KB | SRAM (patch storage, working memory) |
| `$400000 - $40FFFF` | 64 KB | ESP ASIC array (4 ASICs Г— 16 KB each) |
| `$600000 - $600003` | 4 bytes | Key scanner matrix |
| `$600004 - $600005` | 2 bytes | LCD controller (HD44780-compatible) |
| `$FFFF60 - $FFFF9F` | 64 bytes | Timer/counter registers |
| `$FFFFB0 - $FFFFB5` | 6 bytes | MIDI UART (31250 baud) |
| `$FFFFD3 - $FFFFD6` | 4 bytes | I/O ports (LED multiplexing) |
| `$FFFFE0 - $FFFFE8` | 9 bytes | ADC (fader scanning) |

### Boot Sequence

On power-up, the H8S microcontroller:

1. **Loads ROM** вҖ” The firmware contains the MCU operating system, DSP programs, factory presets, and all synthesis parameters
2. **Validates SRAM** вҖ” Checks if battery-backed SRAM contains valid data by searching for known preset names
3. **Factory Reset** (if needed) вҖ” If SRAM is invalid, the firmware runs an automated factory reset sequence that programmatically presses front panel buttons to initialize all user memory
4. **Initializes ESP ASICs** вҖ” Loads DSP programs via the ASIC interface
5. **Starts audio engine** вҖ” Begins processing at 88.2 kHz

The ROM validation searches for specific strings to verify integrity вҖ” the first patch preset name (`"Spit'n Slide Bs "` on the JP-8000, `"Sandstorm       "` on the JP-8080) and the first performance name `"Chariots        "`.

## Roland ESP ASIC

The heart of the JP-8000's sound engine consists of custom **Roland ESP (Embedded Signal Processor)** ASICs вҖ” four in the JP-8000 (8 voices) and five in the JP-8080 (10 voices). These are proprietary fixed-point DSP chips designed by Roland specifically for real-time audio synthesis. No public documentation exists for these processors вҖ” their architecture was reverse-engineered by decapping the physical chips and probing the pins with an Arduino to determine the instruction set and memory organization.

### ASIC Configuration

The JP-8000 uses four ASICs arranged in a pipeline, each processing different stages of the synthesis chain. The JP-8080 adds a fifth ASIC to provide 2 additional voices:

| ASIC | Template | ERAM Size | Description |
|------|----------|-----------|-------------|
| asic0 | `ESP<17>` | 128K words | First processing stage |
| asic1 | `ESP<0>` | None | Second stage (no external memory) |
| asic2 | `ESP<0>` | None | Third stage (no external memory) |
| asic3 | `ESP<19>` | 512K words | Final stage (audio output, delay/effects memory) |
| asic4 | (JP-8080 only) | вҖ” | Fifth ASIC for additional voices |

### Dual-Core Architecture

Each ESP ASIC contains **two processing cores** (Core 0 and Core 1) that execute in parallel:

- **Core 0**: Program memory at PRAM offset `$0000` (768 instruction words)
- **Core 1**: Program memory at PRAM offset `$1000` (768 instruction words)
- Both cores share **GRAM** (Global RAM), **multiplier coefficients**, and **ERAM**
- Each core has its own **IRAM** (256 Г— 32-bit words), **accumulators**, and **program counter**
- Core 1's ERAM transactions are processed; Core 0 executes without ERAM access

At each sample period, both cores execute their full programs (up to 384 instructions each), then synchronize. The IRAM position pointer advances (circular buffer), and ERAM advances one sample position.

### DSP-to-DSP Communication

The ASICs communicate via **GRAM transfers** between sample periods:

```
asic0 вҶ’ asic1:  GRAM[0x80..0x84] вҶ’ GRAM[0x00..0x04]  (3 values)
asic1 вҶ’ asic2:  GRAM[0x80..0x8A] вҶ’ GRAM[0x00..0x0A]  (6 values)
asic2 вҶ’ asic3:  GRAM[0x80..0x8E] вҶ’ GRAM[0x00..0x0E]  (8 values)
                GRAM[0xA0,0xA2]  вҶ’ GRAM[0x20,0x22]    (2 extra values)
```

### Audio Output

Stereo audio is extracted from **asic3**'s GRAM at addresses `0xE8` (left channel) and `0xEC` (right channel) after each sample period.

### Clock and Sample Rate

The ESP ASICs are clocked relative to the H8S microcontroller:

- **MCU clock**: 16 MHz
- **DSP clock**: 16 MHz Г— 1323/625 вүҲ **33.87 MHz** (derived to produce exact 88.2 kHz Г— 384 steps)
- **Sample rate**: **88,200 Hz** (2Г— CD quality)
- **Steps per sample**: 384 (768 total across both cores)
- **ERAM position**: Decrements by 1 each sample (circular buffer)

### MCU вҶ” ASIC Interface

The H8S communicates with the ASICs through a memory-mapped interface at `$400000`:

- **ASIC selection**: Bits 15-14 of the address select which ASIC (0-3)
- **Address within ASIC**: Lower 14 bits
- **Programming mode register**: Address `$2003` within each ASIC selects the write mode

#### Write Modes

| Mode | Code | Description |
|------|------|-------------|
| **Full PRAM write** | `0x54` | Write complete 32-bit instruction words to program memory |
| **Coefficient update** | `0x55` | Update only coefficient and shift fields (hot-patching during playback) |
| **ERAM control update** | `0x56` | Update ERAM address control bits across multiple instructions |
| **PRAM readback** | `0x57` | Read back program memory through readback registers |

The coefficient update mode (`0x55`) is especially important вҖ” it allows the MCU to modify synthesis parameters (filter cutoff, oscillator pitch, envelope levels) in real-time without rewriting entire DSP programs, by patching just the coefficient and shift fields of specific instructions.

#### Readback Registers

Each ASIC provides 4 bytes of readback registers that can be read by the MCU. The DSP writes to these via the `HOST` instruction (opcode `0x34`, mem `0xCA`), enabling DSPвҶ’MCU communication for level meters, voice state, etc.

The ESP uses a **32-bit fixed-point VLIW instruction format** optimized for audio signal processing. Each instruction encodes a multiply-accumulate operation along with memory access and optional control operations.

### Shift Encoding

The 2-bit SHIFT field selects from four predefined shift amounts:

| SHIFT bits | Shift amount | Purpose |
|-----------|-------------|---------|
| `00` | 7 | Standard coefficient scaling |
| `01` | 6 | Slightly higher gain |
| `10` | 5 | Higher gain |
| `11` | 3 | Maximum gain |

### Registers

#### Accumulators

The ESP has two **30-bit signed accumulators** with a 3-stage pipeline:

- **AccA** вҖ” Primary accumulator (default destination)
- **AccB** вҖ” Secondary accumulator

The pipeline delay means that values written to an accumulator are not available for store operations until 3 cycles later. When reading for store operations, the pipeline output can be:

- **Saturated 24-bit**: Clamped to range `[-0x800000, +0x7FFFFF]`
- **Raw 24-bit**: Sign-extended without saturation
- **Raw 30-bit**: Full accumulator width (used for condition testing and ERAM offset)

#### Multiplier Coefficient Registers

Eight 32-bit **multiplier coefficient registers** (`mulcoeffs[0-7]`) provide persistent coefficient storage that can be loaded from the accumulators and used as multiplier inputs. Special indices:
- Index 6: Reads `eramVarOffset << 11` (for variable ERAM addressing)
- Index 7: Aliases to `mulcoeffs[5]` (for coefficient reuse)

#### Condition Skip Field

A 10-bit **skip field** shift register enables conditional execution. When set by the `CMP` instruction, it masks the multiplier input A to zero for subsequent instructions, effectively skipping their computation without branching.

### Memory Spaces

| Space | Size | Access | Description |
|-------|------|--------|-------------|
| **IRAM** | 256 Г— 32-bit | Read/Write per core | Circular buffer for intermediate values |
| **GRAM** | 256 Г— 32-bit | Read/Write shared | Inter-core and inter-ASIC communication |
| **PRAM** | 768 Г— 32-bit per core | Read-only (MCU writes) | Program memory and coefficients |
| **ERAM** | Variable | Read/Write via latch | External memory for delay lines and sample buffers |

#### IRAM Addressing

IRAM uses circular addressing: `effective_address = (MEM + iramPos) & 0xFF`. The `iramPos` pointer decrements by 1 at each sample boundary, creating a sliding window вҖ” new data is always written at the current position while older data remains accessible at higher offsets.

Special MEM values provide hardcoded immediate constants instead of memory access:

| MEM value | Constant | Decimal |
|-----------|----------|---------|
| 1 | `0x000010` | 16 |
| 2 | `0x000400` | 1024 |
| 3 | `0x010000` | 65536 |
| 4 | `0x400000` | 4194304 |

#### ERAM (Extended RAM)

ERAM provides large delay-line memory with a multi-cycle transaction model:

1. **Start** (cycle 0): ERAM control bits in instruction trigger a transaction
2. **Accumulate** (cycles 1-4): Subsequent instructions contribute address offset bits
3. **Commit** (cycle 10): Read or write is performed

ERAM addresses use circular buffering with `eramPos` as the base. A variable offset mode (`eramVarOffset`) enables sample-accurate delay tap positioning. Data is stored in a **lossy 16-bit compressed format** вҖ” a `crunch()` function zeros low-order bits based on the magnitude of the value, preserving more precision for smaller signals.

### Instruction Set Reference

#### Multiply-Accumulate Instructions

The core DSP operation is multiply-accumulate (MAC): `acc += (mulInputA Г— mulInputB) >> shift`

| Opcode | Mnemonic | Operation | Notes |
|--------|----------|-----------|-------|
| `0x00` | **MAC** | `accA += IRAM[mem] Г— coef >> shift` | Accumulate into A |
| `0x04` | **MUL** | `accA = IRAM[mem] Г— coef >> shift` | Clear A, then MAC |
| `0x08` | **STSA.MUL** | Store sat(accA) вҶ’ IRAM[mem]; `accA = sat(accA) Г— coef >> shift` | Store pipeline, use stored value as mul input |
| `0x0C` | **STSB.MUL** | Store sat(accB) вҶ’ IRAM[mem]; `accA = sat(accB) Г— coef >> shift` | Store accB pipeline to memory |
| `0x10` | **MAC.B** | `accB += IRAM[mem] Г— coef >> shift` | Accumulate into B |
| `0x14` | **MUL.B** | `accB = IRAM[mem] Г— coef >> shift` | Clear B, then MAC |

| Opcode | Mnemonic | Operation | Notes |
|--------|----------|-----------|-------|
| `0x30` | **MULC** | `acc += IRAM[mem] Г— mulcoeffs[sel] >> 16` | Complex instruction; COEF field selects coefficient register, sign/negate control, accumulator destination, and clear/accumulate mode |

| Bit(s) | Function |

| 1 | Destination: 0=accA, 1=accB |

| Opcode | Mnemonic | Operation | Notes |
|--------|----------|-----------|-------|
| `0x50` | **CMP** | Set skip field based on accumulator sign | If acc < 0: skipfield \|= 0x3C0; else: skipfield \|= 0x30. Skip field shifts right each cycle, masking mulInputA to zero |

#### Special Operations (Opcode 0x34)

Opcode `0x34` is a polymorphic instruction вҖ” the MEM field selects the actual operation:

**Coefficient Store (MEM = 0xA0-0xAF):**

Stores the saturated pipeline accumulator value into a multiplier coefficient register:
- `mulcoeffs[(MEM >> 1) & 7] = sat(acc)` where `MEM & 1` selects accB (1) or accA (0)

**Control Operations (MEM = 0xC0-0xCF):**

| MEM & 0xF | Mnemonic | Operation |
|-----------|----------|-----------|
| `0x0` | **JZ** | Jump to COEF if accumulator = 0 |
| `0x1` | **JMI** | Jump to COEF if accumulator < 0 |
| `0x2` | **JPL** | Jump to COEF if accumulator > 0 |
| `0x3` | **JMP** | Unconditional jump to COEF |
| `0x4` | **INT** | Set interrupt pins (hardware signal) |
| `0x6` | **DMAC** | Double-precision MAC using previous multiply operands with extended shift |
| `0x7` | **EVAR** | Set ERAM variable offset from accumulator |
| `0xA` | **HOST** | Write saturated accumulator to readback registers (MCU reads these) |
| `0xB` | **ELAT** | Write saturated accumulator to ERAM write latch |
| `0xC-0xF` | **ERDL** | Read ERAM read latch; store to IRAM |

For control operations, MEM bit 5 selects the accumulator (0=accA, 1=accB) and MEM bit 4 enables accumulator clear after the operation.

The **DMAC** (Double-precision MAC) instruction enables chained high-precision multiplication by reusing the previous multiply operands with extended shifting: `mulInputA = prev_mulA >> 7`, `mulInputB = (prev_mulB >> 9) & 0x7F`.

### Execution Model

Every instruction follows the same pipeline:

1. **Fetch**: Read 32-bit instruction from PRAM[PC]; increment PC
2. **Decode**: Extract opcode, MEM, SHIFT, COEF fields
3. **Memory Read**: Read IRAM[mem + iramPos] (or GRAM, or immediate constant)
4. **Execute**: Perform operation-specific side effects (stores, jumps, ERAM control)
5. **Skip Check**: If skip field bit 0 is set, force mulInputA = 0 (conditional skip)
6. **Multiply-Accumulate**: `acc += se<24>(mulInputA) Г— se<8>(mulInputB) >> shift`
7. **Pipeline Advance**: Store current accumulator values into 3-stage pipeline; shift skip field right by 1

The NOP encoding is all-zeros вҖ” it still advances the pipeline but produces no computation.

### JIT Compilation

The Gearmulator emulator includes a **JIT compiler** for the ESP instruction set. Rather than interpreting each instruction, the JIT:

1. Analyzes the ESP program when it changes (triggered by MCU writes to PRAM)
2. Generates optimized native machine code (x86-64 or ARM64 via asmjit)
3. Executes the compiled code directly for each sample period

This is essential for performance since all four ASICs with dual cores must execute 384 Г— 8 = 3,072 instructions per sample at 88,200 Hz, totaling over **270 million ESP instructions per second**.

## Front Panel Interface

### LCD Display

A **2Г—20 character** HD44780-compatible LCD at address `$600004` displays patch names, parameter values, and menus. The emulator intercepts LCD memory changes via callbacks to mirror the display in the plugin UI.

### Buttons

The JP-8000/8080 has **28+ buttons** organized in a multiplexed matrix scanned through the key scanner at address `$600000`. Button types include:

- Oscillator waveform selectors (Osc1, Osc2)
- Filter type and slope (TVF Type, TVF 24dB)
- Performance controls (Upper, Lower, Key Mode, Hold)
- Value Up/Down, Write, Exit
- Numeric keys (1-8) for bank/program selection
- Arpeggiator controls (On/Off, Mode, Range)
- LFO destination and waveform
- Sync, Ring Mod, Portamento, Mono

### Faders

**25+ analog faders** (10-bit resolution, 0-1023 range) are read through the H8S ADC at address `$FFFFE0`:

- Oscillator controls (Range, Pulse Width, PWM Depth, Balance, Cross Mod, Fine Tune)
- Filter (Cutoff, Resonance, Key Follow, Envelope Depth, LFO depths)
- Amplifier (Level, LFO depth)
- Envelopes (Attack, Decay, Sustain, Release for both TVF and TVA)
- Effects (Chorus level, Delay Time/Feedback/Level)
- Tone (Bass, Treble)
- Performance (Tempo, Portamento Time, Pitch Bend, Mod Wheel)
- LFO (Rate, Fade, Depth for both LFOs)

### LEDs

**66 LEDs** are multiplexed through I/O ports at address `$FFFFD3`, indicating active functions, selected modes, and parameter states.

## MIDI Implementation

### Standard MIDI

The JP-8000/8080 uses the H8S UART at **31,250 baud** (standard MIDI rate), mapped at address `$FFFFB0`.

### System Exclusive

### Address Map

| Address (hex) | Contents |
|--------------|----------|
| `00 00 xx xx` | System Parameters |
| `01 00 xx xx` | Temporary Performance (current edit buffer) |
| `02 00 xx xx` | User Patch memory (Bank A: A11-A88, Bank B: B11-B88, 128 total) |
| `03 00 xx xx` | User Performance memory (64 performances) |
| `08 00 xx xx` | Pattern Data (48 patterns, keyboard only) |
| `09 00 xx xx` | Motion Control Data Set A |
| `0A 00 xx xx` | Motion Control Data Set B (rack only) |

### Performance Structure

Each performance contains:

| Offset | Size | Contents |
|--------|------|----------|
| `$0000` | ~36 bytes | Performance Common (name, key mode, split point, voice assign, arpeggiator, tempo) |
| `$0800` | varies | Voice Modulator settings |
| `$1000` | varies | Part Upper settings |
| `$1100` | varies | Part Lower settings |
| `$4000` | 512 bytes | Patch Upper (complete patch data) |
| `$4200` | 512 bytes | Patch Lower (complete patch data) |

### Voice Assignment

The 8 voices can be split between Upper and Lower parts in several configurations:

| Setting | Upper | Lower |
|---------|-------|-------|
| 8-2 | 8 voices | 0 voices |
| 7-3 | 7 voices | 1 voice |
| 6-4 | 6 voices | 2 voices |
| 5-5 | 5 voices | 3 voices |
| 4-6 | 3 voices | 5 voices |
| 3-7 | 2 voices | 6 voices |
| 2-8 | 1 voice | 7 voices |

Key modes: **Single** (one part), **Dual** (layered), **Split** (keyboard split at configurable point).

### MIDI Rate Limiting

The emulator implements MIDI rate limiting at 88,200 Hz sample rate with a **21 ms pause between SysEx patch messages** (per the original hardware's specification) and a minimum threshold of 100 bytes before applying SysEx pausing.

## Emulation Approach

Emulating the JP-8000 presented unique challenges compared to the Motorola DSP56300-based synths:

1. **Unknown DSP Architecture**: Unlike the well-documented DSP56300, no public documentation existed for the ESP ASICs. The instruction set was reverse-engineered by physically decapping the chip die and probing pins with an Arduino.

2. **H8S Microcontroller**: Required implementing a completely separate CPU emulator (H8S instead of MC68k), with its own interrupt system, timers, UART, ADC, and I/O ports.

3. **Four Custom ASICs**: Each ASIC with dual cores means 8 processing cores must be emulated per sample, necessitating JIT compilation for real-time performance.

4. **No Separate Audio Interface**: Unlike the Motorola synths which use standard ESAI/ESSI protocols, the ESP ASICs communicate through custom GRAM transfers that had to be reverse-engineered from firmware behavior.

5. **ERAM Compression**: The lossy 16-bit compression used by ERAM required careful analysis to match the original hardware's quantization characteristics.
